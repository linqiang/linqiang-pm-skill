# Nan Yu 方法论详解

## 背景

Nan Yu 是 Linear 的产品负责人，在硅谷以"痛点驱动"和"极端折中"的产品方法论著称。她的核心理念是：好的产品不是功能的堆砌，而是对用户真实痛苦的精准回应。

## 方法论深度展开

### 痛点挖掘：三层模型

#### 第一层：表层需求（Surface Request）

用户直接表达的内容。通常是功能性的："我需要一个 X"。

**陷阱**：直接按表层需求开发，往往做出"正确但无用"的产品。

#### 第二层：行为真相（Behavioral Truth）

通过观察用户实际操作发现的真相。用户说的和做的往往不一样。

**方法**：
- 观察用户使用现有工具的完整流程
- 记录他们在哪里停顿、重复操作、切换工具
- 注意他们做了但没有抱怨的事情

#### 第三层：情感内核（Emotional Core）

用户在某个瞬间感受到的挫败、焦虑、或无力感。这是产品真正应该解决的东西。

**追问方式**：
- "你上一次用 X 的时候，有没有哪个瞬间觉得特别烦？"
- "如果这件事明天就消失了，你会松一口气还是觉得无所谓？"
- "你有没有因为这个问题放弃过什么？"

### 先试极端，再折中：操作指南

#### 步骤 1：定义极端

对于任何产品决策，先问两个问题：
- **极端 A**：如果我们把这个功能做到极致，完全不考虑成本和限制，最疯狂的版本是什么？
- **极端 B**：如果我们完全不做这个功能，用户会怎样？他们有什么替代方案？

#### 步骤 2：评估两个极端

用三个维度评估：
- **用户价值**：这个版本能消除多少痛苦？
- **实现成本**：需要多少时间、人力、技术复杂度？
- **市场时机**：现在做是早了还是晚了？

#### 步骤 3：寻找平衡点

平衡点不是简单的"中间值"，而是：
- 保留极端 A 中用户最在乎的核心体验
- 用极端 B 的简洁性控制复杂度
- 找到用最小投入撬动最大用户价值的那个点

#### 典型应用场景

| 场景 | 极端 A | 极端 B | 平衡点思路 |
|------|--------|--------|------------|
| 通知系统 | 所有事件实时推送 | 零通知，用户主动查看 | 只推送用户定义的关键事件 |
| 权限管理 | 细粒度到字段级别 | 只有管理员/成员两个角色 | 3-5 个预设角色 + 关键资源自定义 |
| 搜索功能 | 全文搜索 + AI 语义理解 | 只支持精确匹配 | 模糊匹配 + 最近使用优先排序 |

### 无意识苦差事：发现方法

用户不会主动告诉你这些事情，因为他们已经习惯了。发现方法：

1. **影子观察**：跟着用户看他们完整的工作流程，记录每一个"小动作"
2. **时间审计**：让用户记录一周的工作时间分配，看哪些重复操作占了不成比例的时间
3. **工具切换计数**：数一数用户在完成一个任务时需要切换多少个工具
4. **复制粘贴追踪**：频繁的复制粘贴几乎总是意味着有一个自动化机会

**Linear 案例**：Linear 早期发现开发者在任务管理中最痛苦的不是"功能不够"，而是"操作太慢"。所以 Linear 把性能和键盘操作做到极致，而不是堆砌更多功能。

### 快速验证框架

#### 验证优先级矩阵

| | 高不确定性 | 低不确定性 |
|---|---|---|
| **高影响** | 必须验证，用最小实验 | 直接做，快速上线 |
| **低影响** | 暂不做，观察 | 积压，有空再说 |

#### 最小实验设计

1. **假设陈述**：我们相信 [用户群] 在 [场景] 下面临 [痛点]，如果我们提供 [方案]，他们会 [行为改变]
2. **成功标准**：定义具体可测量的指标和阈值
3. **实验边界**：最多花多少时间、涉及多少用户、用什么方式实现
4. **学习计划**：无论成功失败，我们要学到什么

### PM 即 GTM：产品决策清单

每个功能决策前回答：

- [ ] 用户第一次见到这个功能时，能在 5 秒内理解它做什么吗？
- [ ] 现有用户会自然发现这个功能，还是需要额外引导？
- [ ] 用户用了这个功能后，有理由告诉别人吗？
- [ ] 这个功能的名字，用户会用自己的话怎么描述？
- [ ] 如果只能在发布时写一句话介绍，那句话是什么？

### 结构化复盘：操作指南

复盘不是"成功汇报"，是对假设的诚实审判。

#### 复盘时机

| 时机 | 适用场景 |
|------|---------|
| 功能上线后 1-2 周 | 新功能发布，数据和反馈初步积累 |
| 迭代周期结束 | Sprint 或 cycle 结束时 |
| 指标异动时 | 关键指标突然上升或下降 |
| 方向决策前 | 在做重大 pivot 或资源调整之前 |

#### 复盘四步法

**第一步：还原假设**

把当初做决策时的核心假设原原本本列出来，不要事后美化。

- 我们当时相信什么？
- 我们用什么依据做的判断？
- 成功标准是什么？

**第二步：对照事实**

用数据和用户反馈逐条验证假设。

| 假设 | 预期结果 | 实际结果 | 判定 |
|------|---------|---------|------|
| 假设 1 | ... | ... | 验证 / 推翻 / 部分验证 |
| 假设 2 | ... | ... | ... |

关键原则：
- 数据说话，不要用"感觉还行"替代具体数字
- 区分"指标达标"和"用户真的满意"——数字好不等于体验好
- 如果没有数据，这本身就是一个发现：说明验证环节有缺失

**第三步：提炼意外**

最有价值的往往不是预期内的结果，而是意料之外的发现。

- 用户有没有用我们没想到的方式在用？
- 有没有我们以为不重要、但用户特别在乎的细节？
- 有没有我们做了但用户完全没注意到的功能？
- 竞品或市场有没有出现我们没预料到的变化？

**第四步：明确行动**

复盘必须落地为具体行动，否则就是浪费时间。

每个行动项必须回答：
- **做什么**：一句话说清楚
- **为什么现在做**：基于哪条复盘发现
- **谁负责**：明确 owner
- **怎么验证**：这个调整的成功标准是什么

行动分三类：
- **加倍投入**：假设验证了，效果好于预期，继续加码
- **调整方向**：假设部分验证，需要修正路径
- **果断止损**：假设被推翻，停止投入，释放资源

#### 复盘反模式

- **成功归因于自己，失败归因于环境**：复盘的价值在于诚实
- **只看结果不看过程**：好结果可能是运气，坏结果可能方法是对的但时机不对
- **复盘变成批斗会**：目标是学习，不是追责
- **复盘了但不改**：如果复盘结论没有变成具体行动，下次别复盘了
- **过度复盘**：不是所有事情都值得正式复盘，小事快速回顾即可

## 反模式

这些是 Nan Yu 方法论明确反对的做法：

- **功能清单驱动**：把竞品功能列表当作产品路线图
- **假设即事实**：没有验证就全力投入
- **完美主义陷阱**：追求完美版本而迟迟不上线
- **指标崇拜**：只看数字不看用户情感
- **共识优先**：为了团队和谐而回避真实判断
